# Product Reset Research Memo

Last updated: 2026-04-30

This memo satisfies Phase 1 of the product reset plan. It is intentionally
method-first. No new UI should be built from the rejected workbench until
the product flow below is represented in data payloads, experiments, and
validation outputs.

## Executive Decision

The deployed app is technically operational, but it is not the target
product. It proves that FastAPI, React, derived assets, bilingual UI,
theme switching, and VPS deployment work. It does not yet prove a
scientific workflow.

The reset must rebuild the product around one methodological sequence:

1. choose a dataset family and its supervision state
2. inspect evidence that actually exists
3. choose a corpus recipe
4. define alphabet, word, document, corpus, topic, and topic mixture
5. run or display PTM/LDA outputs only after the corpus is explicit
6. compare against appropriate non-topic baselines
7. show inference only when labels or measurements exist
8. show validation and caveats before making claims

The public app must never imply that a topic is a mineral, class,
endmember, or response variable unless that claim is supported by labels,
measurements, spectral-library alignment, or a validation panel.

- generic right inspector that does not define the current method

  are explicit

Immediate product implication:

- production is a checkpoint, not the design baseline
- the next branch should add research, data taxonomy, corpus metadata,
  experiments, and validation assets before touching the UI again
- raw local scientific validation is primary; compact web assets are a
  later projection of that work

## Method Mapping

The app must teach and enforce this mapping:

| Term | Product meaning | Required metadata |
|---|---|---|
| Alphabet | finite set of spectral symbols a tokenizer can emit | band centers, units, normalization, quantization, bad-band mask |
| Word | one emitted symbol or weighted spectral event | token id, token family, physical meaning, source band/region |
| Document | one modelled object: spectrum, pixel, patch, ROI, class, superpixel, sample support, or measurement window | dataset id, sample id, geometry, supervision, source spectra count |
| Corpus | all comparable documents generated by one recipe | recipe id, vocabulary id, document count, vocabulary size, length distribution |
| Topic | probability distribution over words | model id, topic-word weights, seed, hyperparameters |
| Topic mixture | probability distribution over topics for a document | document id, topic vector, inference method, model id |
| Inference feature | topic mixture or derived topic statistic used in a downstream model | split id, target, baseline, metric, caveat |

The app should not show a topic map unless the corpus and topic-mixture
definitions are already visible. The app should not show a cluster map
unless the feature space, spatial use, supervision use, and caveat are
visible.

## Research Anchors

### LDA / PTM

Blei, Ng, and Jordan define LDA as a generative probabilistic model for a
corpus of discrete documents, where documents are mixtures of topics and
topics are distributions over words. For this product, that means spectra
must be discretized into words before LDA can mean anything scientific.

Actionable rule:

- every LDA output in the app must expose the count matrix recipe, not
  just the fitted topic result

Source:

- <https://jmlr.org/papers/v3/blei03a.html>

### Spectral LDA For HSI/MSI

Wahabzada et al. use LDA to uncover spectral characteristics of drought
stress in plant hyperspectral data. The important product lesson is that
topic modelling can be positioned as discovery of recurring spectral
patterns, but only when the mapping from spectra to discrete observations
is documented.

Actionable rule:

- vegetation/MSI workflows should compare topic outputs against known
  indices such as NDVI and against label or stress variables when
  available

Source:

- <https://arxiv.org/abs/1210.4919>

### PM-LDA And Endmember Variability

Zou and Zare apply Partial Membership LDA to hyperspectral unmixing with
endmember variability. This is closer to the mineral/clay goal than a
generic topic demo because it treats mixed membership and variability as
central rather than incidental.

Actionable rule:

- mineral/unmixing workflows should include PM-LDA/NMF/unmixing
  comparisons instead of treating standard LDA as the only topic method

Sources:

- <https://arxiv.org/abs/1609.03500>
- <https://arxiv.org/abs/1703.06151>

### Spectral Variability

Borsoi et al. review spectral variability in hyperspectral unmixing. This
supports the product thesis: variability is not disposable noise, but it
also warns that variability can come from artifacts, acquisition
conditions, and physical heterogeneity.

Actionable rule:

- every strong interpretation must separate material/process variability
  from illumination, calibration, atmosphere, noise, and sensor artifacts

Source:

- <https://arxiv.org/abs/2001.07307>

### Superpixels And Spatial Documents

SLIC is a compact spatial grouping baseline. It is useful because region
documents are central to the reset thesis: a document can be a
superpixel, patch, ROI, class, or measured support rather than one
spectrum.

Actionable rule:

- SLIC should be used as a document-boundary experiment and a spatial
  baseline, not as decoration

Sources:

- <https://scikit-image.org/docs/stable/api/skimage.segmentation.html>
- <https://www.iro.umontreal.ca/~mignotte/IFT6150/Articles/SLIC_Superpixels.pdf>

### Spectral Libraries

USGS Spectral Library v7 is already partially represented locally. It is
appropriate for Family A individual spectra and for spectral-reference
alignment, but only compact curated slices should be committed.

ECOSTRESS/JPL is a high-value extension for mineral, vegetation, soil,
water, and man-made materials. The public download surface is
reproducible and exposes category counts, but the current bulk checkout
flow resolves to a login form. That means the repo can now version
access metadata, but still cannot claim a reproducible anonymous export
path for compact slices.

Sources:

- <https://www.usgs.gov/data/usgs-spectral-library-version-7-data>
- <https://speclib.jpl.nasa.gov/>

### HIDSAG

HIDSAG is the strongest candidate for Family D because it provides
sample-level geometallurgical response variables alongside VNIR/SWIR
spectral data. Its role is not "another spectral library"; it is the
first serious route to supervised regression/classification over region
or sample documents. Local raw coverage now includes:

- `GEOMET.zip`: 146 samples, 5 geometallurgical variables, 438 HSI cubes
- `MINERAL1.zip`: 99 samples, 33 mineral abundance variables, 297 HSI cubes
- `MINERAL2.zip`: 20 samples, 25 mineral abundance variables, 60 HSI cubes
- `GEOCHEM.zip`: 28 samples, 18 geochemical variables, 106 measurement
  supports, 318 HSI cubes
- `PORPHYRY.zip`: 28 samples, 10 mineral abundance variables plus one
  categorical ore-group field, 56 measurement supports, 168 HSI cubes

A compact derived export is now versioned with mean and
standard-deviation spectra per cube so the repo can inspect real HIDSAG
signals without shipping the raw ZIPs. A second compact artifact now
stores `3 x 3` fixed-grid patch-region documents per measurement
support for local hierarchical validation. The curated export now also
preserves per-modality wavelength vectors directly from the HIDSAG HDF5
attributes.

Actionable rule:

- HIDSAG should enter only as a curated subset with model cards, response
  variables, train/test splits, and explicit licensing/provenance notes

Current local benchmark reading:

- the local core now benchmarks `MINERAL1`, `MINERAL2`, `GEOMET`,
  `GEOCHEM`, and `PORPHYRY`
- `MINERAL1` now demonstrates why naive five-fold validation was too
  optimistic: under process-aware `P1/P2/P3` group splits, several
  earlier strong scores collapse materially
- `MINERAL2` remains sample-poor, but patch-region topic mixtures now
  produce slightly better results than the older topic pass for targets
  such as Phengite and Quartz
- `GEOMET` still favors raw/PLS baselines, but the patch-region topic
  model now activates `4/6` topics instead of collapsing almost
  completely
- `GEOCHEM` validates the multi-measurement case: routed or region-topic
  regressors already reach positive `R^2` on Fe, Ca, S, and Cu
- `PORPHYRY` is now local and benchmarked under ore-group-aware splits,
  but its current mineral regression signal is still weak and mostly
  negative
- raw or PCA-compressed spectra are still stronger for most balanced
  binary classification tasks
- topic-routed, cube-topic, and region-topic local models are not yet
  defendable as a finished method because topic activity remains uneven
  across subsets
- a first heuristic bad-band policy is now versioned separately from the
  main benchmark, along with a preprocessing-sensitivity pass over all
  five subsets
- that sensitivity pass shows two things at once:
  - preprocessing choice can materially change downstream results,
    including large gains on selected `MINERAL2`, `GEOMET`, `GEOCHEM`,
    and `PORPHYRY` targets
  - those gains do not automatically validate topic routing, because
    some of the strongest improvements still come from raw/PLS-style
    baselines while topic activity remains partially collapsed

Source:

- <https://www.nature.com/articles/s41597-023-02061-x>

- explain one-spectrum documents

- support supervised models only when labels or response variables exist

| Dataset | Supervision | Main blocker | Decision |

## Validation Plan

### Corpus Integrity

For each recipe:

- vocabulary size
- document count
- document length distribution
- zero-token document count
- token coverage by band or spectral region
- corpus file size
- deterministic output under fixed seed

Failure conditions:

- token meaning cannot be reversed from the manifest
- document geometry is missing for region documents
- vocabulary grows without an explicit sparsity strategy

### Topic Stability

Run repeated LDA/PTM fits across seeds and match topics by cosine or
Jensen-Shannon similarity over topic-word distributions.

Report:

- mean matched-topic similarity
- unstable topics
- topic-mixture variance by document family
- sensitivity to number of topics

Failure conditions:

- topics reorder or collapse without stable matching
- a visualization uses one seed as if it were definitive

### Quantization Sensitivity

Compare bin counts such as 4, 8, 12, and 16, and compare uniform versus
quantile binning when appropriate.

Report:

- vocabulary size
- document length
- topic stability
- label alignment when labels exist
- nearest-library consistency when references exist

Failure conditions:

- topic interpretation changes drastically with minor bin changes
- quantile bins are shown as physically meaningful thresholds

### Document Definition Sensitivity

Compare pixel, class/ROI, fixed patch, SLIC superpixel, and measurement
support documents.

Report:

- label purity for labeled scenes
- spatial coherence for image scenes
- class/topic association
- topic map boundary behavior
- document length and spectra count distributions

Failure conditions:

- a region method is used without reporting how regions were made
- topic semantics are compared across incompatible document definitions

### Label And Measurement Association

For Family B and D:

- label-topic association matrix
- ARI/NMI between topic-dominant labels and ground truth where meaningful
- classification metrics from topic mixtures
- regression metrics for response variables
- per-topic errors and failure cases

Failure conditions:

- inference UI appears for unlabeled/unmeasured data
- no train/test split is reported

### Spectral-Library Alignment

For Family A and C:

- top-k nearest references by SAM/cosine/distance
- agreement across raw spectra, normalized spectra, and topic-derived
  profiles
- absorption-region notes when wavelengths are calibrated

Failure conditions:

- nearest reference is called a material ID
- approximate wavelength axes are used for absorption claims

### Baseline Comparison

Each dataset/recipe result should name a comparison purpose:

- SLIC: did spatial grouping improve document quality?
- KMeans/GMM/hierarchical: do non-topic spectral groups explain the same
  structure?
- SAM: do spectral references support or contradict topic interpretations?
- NMF/unmixing: does mixture modelling explain the scene better than
  topics for unmixing-style data?
- raw/PCA supervised models: do topic mixtures add predictive value?

Failure conditions:

- scatter plots without feature-space metadata
- baselines shown only because they look interesting
